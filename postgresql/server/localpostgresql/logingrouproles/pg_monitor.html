pg_monitor#

What is pg_monitor?#

pg_monitor allows a role to view PostgreSQL performance, activity, and statistics ‚Äî without modifying data.

Think of it as:

Read-only access to the database internals

What pg_monitor CAN do#

With pg_monitor, a role can:

See activity

  • pg_stat_activity

  • Active queries

  • Long-running queries

  • Connection counts

  • Backend states (idle, active, waiting)

See performance stats

  • pg_stat_database

  • pg_stat_user_tables

  • pg_stat_user_indexes

  • Buffer hits

  • I/O statistics

  • Index usage

See locks & waits

  • pg_locks

  • Blocking queries

  • Deadlock diagnosis

See configuration (read-only)

  • Some server settings via:

  • pg_settings

  • pg_stat_bgwriter

‚ùå What pg_monitor CANNOT do

  • ‚ùå Read table data

  • ‚ùå Modify data

  • ‚ùå Execute commands

  • ‚ùå Kill backends

  • ‚ùå Access filesystem

  • ‚ùå Execute OS programs

It is observational only

Why PostgreSQL created pg_monitor#

Before pg_monitor, DBAs had two bad choices:

  • Grant superuser ‚ùå (too dangerous)

  • Grant nothing ‚ùå (no visibility)

pg_monitor solves this:

  • ‚úî Safe

  • ‚úî Auditable

  • ‚úî Least-privilege

  • ‚úî Perfect for dashboards & tooling

Internally, pg_monitor includes

It automatically grants:

  • pg_read_all_settings

  • pg_read_all_stats

So you don‚Äôt need to grant those individually.

Example: Who should get pg_monitor?

‚úÖ Good candidates

  • Monitoring tools (Prometheus exporters)

  • Read-only DBA accounts

  • Performance dashboards (Grafana)

  • SRE / Ops teams

  • Support engineers

‚ùå Bad candidates

  • Application users

  • Web apps

  • ETL users

  • Untrusted users

Example Queries

SELECT * FROM pg_stat_user_tables;

SELECT * FROM pg_locks;

All of these work with pg_monitor.

How to grant it#

‚úî Reversible
